# Alpha 1.0 Design: Dev Infrastructure & Quality Gates

**Date:** 2026-03-04
**Status:** Approved
**Author:** Claude + Tom

## Context

Idle is a fork of Happy Engineering (slopus/happy) — a mobile/web client for Claude Code. After completing Phase 1 (fork, rebrand, deploy) and Phase 1.5 (security hardening, documentation), we hit serious friction during mobile E2E testing:

- PGlite Bytes serialization bug (Uint8Array vs Buffer) caused 500 errors on session/machine creation
- Auth persistence used sessionStorage (dies on tab close)
- No way to test CLI→Server→WebApp flow without deploying to VPS and testing manually
- No regression tests for any of the bugs found
- Deploy-test cycle is ~3 minutes per iteration
- Only server and CLI tests run in CI; app/agent/wire tests are invisible

**Bottom line:** We're building features but can't verify they work without manual end-to-end testing against production. This is unsustainable for Alpha.

## Decision

Build the dev infrastructure and quality gates needed to develop, test, and iterate on Idle efficiently. Four workstreams:

### 1. Dev Environment

**Architecture:**
```
Mac (dev)                                  IONOS VPS (always-on)
├── yarn web → localhost:8081 (hot reload)  idle-api.northglass.io (server)
│   └── API calls go to VPS server ────►   ├── Fastify + PGlite
├── idle-coder (local build) ──────────►   ├── Socket.IO WebSocket
└── git push → CI deploys                  └── idle.northglass.io (prod web app)
```

**Key insight:** `yarn web` already defaults to `idle-api.northglass.io` — no config changes needed. The app resolves server URL via `serverConfig.ts` with fallback chain: user setting → `EXPO_PUBLIC_IDLE_SERVER_URL` env var → `https://idle-api.northglass.io`.

**Changes:**
- **Quick-deploy script** (`scripts/deploy-server-quick.sh`): Run typecheck + tests locally, then SSH deploy directly. Bypasses CI for iteration speed (~30s vs ~3min).
- **`.env.example` files** for all packages: Document required/optional config so new sessions don't reverse-engineer it.
- **Root Makefile or npm scripts**: `yarn dev:web` (hot reload against VPS), `yarn deploy:server:quick` (fast deploy).

### 2. Test Infrastructure

**All packages tested in CI:**

New `test-all.yml` workflow runs on every push/PR:
- `idle-wire` (2 test files — fast, foundational, must pass first)
- `idle-server` (5 test files — API routes, storage)
- `idle-cli` (45 test files — API, encryption, daemon, ACP)
- `idle-app` (24 test files — components, sync, auth)
- `idle-agent` (9 test files — API, auth, encryption)

Build order matters: wire → server/cli/app/agent (wire is a dependency).

**Regression tests for this session's bugs:**
- PGlite Bytes field serialization (Buffer vs Uint8Array)
- Auth token persistence (localStorage vs sessionStorage on web)
- Logout redirect (navigate to '/' vs reload)

### 3. E2E Test Suite

**Full CLI→Server→WebApp tests** in `packages/idle-e2e/`.

**Framework:** Vitest + Playwright + child_process (spawn CLI).

**Test scenarios:**
1. CLI auth login → browser accepts → CLI receives credentials
2. CLI registers machine → server returns 200 → machine visible in web app
3. CLI starts Claude session → web app shows session in session list
4. Web app clicks session → terminal connect page loads → WebSocket connects
5. CLI sends message → web app renders it in terminal view
6. Web app sends reply → CLI receives it
7. Sign out in web app → redirects to home, auth state cleared
8. CLI auth logout → token invalidated on server

**Architecture:**
```
Vitest Test Runner
├── Playwright browser → localhost:8081 (local Expo web dev server)
├── Child process → idle-coder CLI (local build from source)
├── Both talk to → idle-api.northglass.io (VPS server)
└── Assertions on both: CLI stdout/stderr + browser DOM state
```

**Test isolation:** Each test creates a fresh account, cleans up after. Tests can run in parallel if they use different accounts.

**CI integration:** Runs after `deploy-server.yml` succeeds, or manually triggered. Not on every push (too slow, depends on VPS).

### 4. Documentation & Quality Gates

**ADRs to create:**
- ADR-002: PGlite as embedded database
- ADR-003: Deployment architecture (VPS + Cloudflare)
- ADR-004: E2E encryption model
- ADR-005: Dev/test strategy

**New docs:**
- `CONTRIBUTING.md`: Setup guide for new Claude sessions and contributors
- `.env.example` files for all packages
- Updated `ROADMAP.md` with Alpha 1.0 milestones

**Quality gates enforced (per global CLAUDE.md):**
1. Code runs without errors
2. All tests pass (run them, show output)
3. New functionality has tests
4. No hardcoded secrets
5. Major decisions captured in ADRs
6. Conventional commits
7. Navigation docs updated

### 5. Upstream Compatibility

**Strategy:** Minimize modifications to upstream files. Our changes are:

**Modified upstream files (potential merge conflicts):**
- `packages/idle-app/sources/auth/tokenStorage.ts` (sessionStorage → localStorage)
- `packages/idle-app/sources/auth/AuthContext.tsx` (logout redirect)
- `packages/idle-app/sources/components/AgentInput.tsx` (hide default labels)
- `packages/idle-app/sources/components/TabBar.tsx` (PWA padding)
- `packages/idle-server/sources/app/api/routes/sessionRoutes.ts` (Buffer fix)
- `packages/idle-server/sources/app/api/routes/machinesRoutes.ts` (Buffer fix)
- `packages/idle-server/sources/app/kv/kvMutate.ts` (Buffer fix)

**Idle-only additions (no conflict risk):**
- `docs/`, `CLAUDE.md`, `AGENTS.md`, `.github/workflows/`
- `packages/idle-e2e/` (new package)
- `CONTRIBUTING.md`, `.env.example` files
- `scripts/deploy-server-quick.sh`

**Process:** Monthly `git fetch upstream && git merge upstream/main`. Run E2E suite after merge to verify. Document conflicts in CONTRIBUTING.md.

## Consequences

**Positive:**
- Can iterate on web app UI instantly (hot reload)
- Server bugs caught before deploy (regression tests)
- Full stack verified automatically (E2E suite)
- New Claude sessions productive in minutes (CONTRIBUTING.md + .env.example)
- Upstream changes mergeable with known conflict points documented

**Negative:**
- E2E tests depend on VPS being healthy (not fully offline-capable)
- More CI minutes (test-all.yml runs on every push)
- Initial setup effort before we can ship new features

**Trade-off accepted:** We're investing in infrastructure now to move faster later. The alternative (manual testing against production) doesn't scale.
