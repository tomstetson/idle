# Idle v1.5 Comprehensive Build Design

**Date:** 2026-03-07
**Status:** Approved
**Goal:** Get Idle to a state where CLI + iOS app workflow is reliable for daily use, with invited testers via Cloudflare Access email-gate and CLI via npm pre-release.

## Context

Idle is a fork of Happy Engineering (slopus/happy), rebranded under Northglass LLC. As of this design:

- **Idle is 55 commits ahead of upstream** ‚Äî no merge needed. Cherry-pick specific features if desired.
- Phases 1 and 1.5 are complete: server deployed, web app deployed, CI/CD pipelines working, E2E encryption verified, auth hardening done, docs and ADRs written.
- Apple Developer account just approved. Expo org `northglass` created on expo.dev.
- No TestFlight build exists yet.

## Decisions

- **Platform priority:** CLI > iOS (TestFlight) > Server > Web (maintain only)
- **Target audience:** Developer (Tom) + invited testers via email-gate
- **No upstream merge:** We're ahead; cherry-pick if needed
- **CLI published as pre-release:** `idle-coder@0.15.0-alpha.1` on npm
- **Public access:** Cloudflare Access email-gate (not fully open)
- **Voice features:** Deferred past v1.5

## Milestone 3: Permission Handling + Session Resume

**Work needed:** End-to-end validation on TestFlight. Fix any gaps found during real-device testing. This is validation, not new feature work.

**Effort:** 4-6 hours (2h display + 3-4h swipe gesture)

### Dynamic Slash Command Discovery (MEDIUM effort ‚Äî 50% done)

**What exists:** CLI extracts `slashCommands` from SDK metadata on session start via `metadataExtractor.ts`. App has fuzzy autocomplete in `useAutocompleteSession.ts`. Static ignore list + default commands in `suggestionCommands.ts`.

**What's missing:** Command descriptions not extracted. Hardcoded ignore list lags upstream changes.

**Work needed:**
- Extend metadata extraction to include command descriptions (if SDK provides them)
- Replace static ignore list with dynamic list from CLI
- Update app autocomplete to display descriptions alongside command names

**Key files:**
- `packages/idle-cli/src/claude/sdk/metadataExtractor.ts` ‚Äî metadata extraction
- `packages/idle-app/sources/sync/suggestionCommands.ts` ‚Äî static list + search
- `packages/idle-app/sources/hooks/useAutocompleteSession.ts` ‚Äî autocomplete hook

**Effort:** 2-3 days

### Terminal Re-ordering & Grouping by Machine (MEDIUM effort ‚Äî 0% done)

**What exists:** `project-group` type in `SessionListViewItem` schema (unused). Machine data (host, platform, OS) available on all sessions. `ActiveSessionsGroup.tsx` shows machine info for active sessions only.

**Work needed:**
- Activate machine grouping in `buildSessionListViewData()` ‚Äî group sessions by `machineId`
- Render collapsible machine headers using Item component
- Add pin/reorder via long-press context menu (store `pinnedSessions` in local settings)

**Key files:**
- `packages/idle-app/sources/sync/storage.ts` ‚Äî `buildSessionListViewData()` (lines 152-245)
- `packages/idle-app/sources/components/ActiveSessionsGroup.tsx` ‚Äî active session display
- `packages/idle-app/sources/components/SessionsList.tsx` ‚Äî main session list

**Effort:** 3-5 days

### Edit Already-Sent Messages (HIGH effort, HIGH risk ‚Äî 0% done)

**What exists:** Nothing. No wire types, no SDK support confirmed.

**Approach:** Investigate Claude Code SDK capability early in M4. Make go/no-go decision.
- If SDK supports editing natively: new wire types (`sessionEditEvent`), CLI handler, app UI (long-press to edit)
- If SDK doesn't support it: evaluate `--resume` workaround or defer to backlog

**Key unknowns:**
- Does Claude Code SDK support message editing?
- Does editing require re-running the entire turn from that point?

**Effort:** 2-4 days investigation. If go: 2-3 weeks full implementation across 4 packages.

### Push Notifications (MEDIUM effort ‚Äî 30% done)

**What exists:** Server has `POST /v1/push-tokens` endpoint, database stores tokens in `AccountPushToken` table, `expo-notifications` in app dependencies, notification handler configured in root `_layout.tsx`.

**What's missing:** No push *sending* logic. No app-side token registration flow. No notification payload structure.

**Work needed:**
- App: Request notification permission, register push token via `getExpoPushTokenAsync()`, call `registerPushToken()` in auth flow
- Server: Send pushes via Expo Push Service (handles APNs) when new messages arrive. Trigger in message update handler. Handle expired tokens.
- Payload: `{title: "Session Name", body: truncate(messageText, 100), badge: 1}`

**Key files:**
- `packages/idle-server/sources/app/api/routes/pushRoutes.ts` ‚Äî token registration (lines 7-48)
- `packages/idle-app/sources/sync/apiPush.ts` ‚Äî client-side registration helper
- `packages/idle-app/sources/app/_layout.tsx` ‚Äî notification handler setup

**Effort:** 8-12 hours

### Cross-Machine Session Mobility (MEDIUM-LARGE ‚Äî needs audit)

**Current state:** Sessions tied to machines via `AccessKey` model with `accountId_machineId_sessionId` unique constraint. Encryption model (per SECURITY.md) describes session keys wrapped in user's public key ‚Äî which may already support cross-machine access.

**Approach:**
1. **Audit** (2 hours): Verify whether session encryption keys are user-encrypted or machine-encrypted
2. **If user-encrypted (likely):** Add handoff endpoint `POST /v1/sessions/{sessionId}/handoff` to create AccessKey for new machine. Small effort (4-6 hours).
3. **If machine-encrypted:** Re-encryption logic required. Large effort (10-14 hours).

**Key files:**
- `packages/idle-server/prisma/schema.prisma` ‚Äî AccessKey model (lines 280-297)
- `docs/SECURITY.md` ‚Äî encryption model documentation
- `packages/idle-server/sources/app/api/routes/` ‚Äî add handoff endpoint

**Effort:** 2h audit + 4-14h implementation depending on findings
